# Developer Persona

## Role
Software Developer or Software Engineer. Writes code, builds features, debugs problems, and ships software. Cares about tools, APIs, and frameworks that help them do their job better.

## Priorities
- Code quality and maintainability
- API design and developer experience
- Documentation quality, especially getting-started guides
- Build and test tooling
- Debugging and troubleshooting capabilities
- Community support and ecosystem health
- Performance characteristics of libraries and frameworks

## Reading Patterns
- Jumps straight to code examples and quickstart guides.
- Evaluates API ergonomics from the first code snippet they see.
- Checks GitHub stars, recent commit activity, and open issues.
- Looks for compatibility with their existing stack.
- Reads error messages and debugging sections closely.

## Common Concerns
- "Can I see a working code example in the first two minutes?"
- "How good is the documentation? Is it kept up to date?"
- "What does the error handling look like?"
- "Is there an active community? Can I get help when I am stuck?"
- "Does this play well with my existing tools and frameworks?"
- "What is the testing story? Can I write unit tests easily?"
- "How much boilerplate do I have to write?"

## What Makes Them Stop Reading
- No code examples
- Marketing language instead of technical content
- Outdated or incomplete documentation
- Complex setup with no quickstart path
- Ignoring developer experience concerns

## What Wins Them Over
- Clean, working code examples that they can copy and run
- Excellent documentation with real-world usage patterns
- Active community and responsive maintainers
- Good error messages and debugging tools
- Minimal boilerplate and sensible defaults
